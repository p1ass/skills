# Code Reviewer

Google's Code Review Guidelines に基づき、段階的にコードレビューを実施する。

レビューの目的は「コードベースの健全性を時間とともに改善すること」にある。完璧なコードを求めるのではなく、変更がコードベース全体の品質を確実に向上させているかを検証する。

## ワークフロー

### Step 1: コンテキストの把握

レビュー対象を特定し、変更の背景を理解する。背景を理解せずに差分だけ見ると、設計意図を見落とす。

1. PR番号やブランチ名から対象を特定する
2. PRの説明文、関連するIssue・チケットがあれば読む
3. `git diff` または `gh pr diff` で差分を取得する
4. 変更されたファイルの一覧と変更量を俯瞰する

```bash
# PRの差分を取得
gh pr diff <PR番号>

# ブランチ間の差分を取得
git diff <base-branch>...HEAD --stat
git diff <base-branch>...HEAD
```

### Step 2: 設計レビュー（マクロ視点）

最初に「そもそもこのアプローチで正しいか」を判断する。行単位の指摘より先に設計の妥当性を確認する理由は、設計に問題があれば詳細レビューの大半が無駄になるため。

確認する観点:
- **変更の必要性**: この変更は要件を満たしているか？
- **設計の適切性**: システム全体のアーキテクチャに適合しているか？
- **複雑度**: 必要以上に複雑な実装になっていないか？ もっと単純な方法はないか？
- **Over-engineering**: 現在の要件に対して過剰に汎用的な設計になっていないか？

設計に根本的な問題がある場合は、詳細レビューに進む前にその旨を指摘する。

### Step 3: 詳細レビュー（ミクロ視点）

設計に問題がなければ、変更された各ファイルを読み、以下の観点で確認する。`references/review-guideline.md` のチェックリストを参照しながら進める。Goプロジェクトの場合は `references/go-review-guideline.md` も併せて参照する。

#### 3a. 正確性とエッジケース

コードが意図した通りに動作するかを確認する。

- 境界値や例外的な入力に対する処理は適切か？
- エラーハンドリングは言語の慣習に沿っているか？ エラーを握りつぶしていないか？
- nil/null、空文字列、空配列などのエッジケースは考慮されているか？
- 並行処理がある場合、競合状態やデッドロックの可能性はないか？

#### 3b. セキュリティ

外部入力を扱うコードや認証・認可に関わる変更は重点的に確認する。

- 外部入力のバリデーションは適切か？
- SQLインジェクション、XSSなどの脆弱性はないか？
- 機密情報がログやエラーメッセージに含まれていないか？
- 認証・認可のチェックが適切に実施されているか？

#### 3c. パフォーマンス

明らかなパフォーマンス問題がないか確認する。微細な最適化よりも、O(n²)のループやN+1クエリなどの構造的な問題に注目する。

#### 3d. 一貫性と可読性

既存のコードベースとの一貫性を確認する。

- 命名規則は既存コードと統一されているか？
- 設計パターンやエラーハンドリングのパターンはプロジェクトの慣習に沿っているか？
- コメントは「なぜ」を説明しているか？（「何を」の説明は不要）

#### 3e. テスト

テストコードが変更に伴って適切に追加・更新されているか確認する。

- 重要な振る舞いに対するテストが含まれているか？
- エッジケースやエラーケースのテストはあるか？
- テストは実装の内部詳細ではなく、振る舞いを検証しているか？

### Step 4: レビュー結果の出力

指摘を重要度順に整理して出力する。指摘が多い場合でも、設計上の問題 → バグ → その他の順に優先度をつけ、重要な問題を先に伝える。

#### 重要度の順序

1. **設計上の問題** — 最優先。アプローチ自体の再考が必要
2. **機能の不具合** — バグやエッジケースの見落とし
3. **セキュリティ** — 脆弱性や情報漏洩のリスク
4. **パフォーマンス** — 構造的なボトルネック
5. **可読性** — コードの理解しやすさ
6. **スタイル** — 一貫性の問題

#### ラベル

各指摘の先頭にラベルを付与し、対応の緊急度を明示する。

| ラベル | 用途 |
|---|---|
| **must** | 修正必須。マージ前に対応が必要（バグ、セキュリティ、設計上の欠陥） |
| **should** | 修正推奨。対応が望ましいが、理由があれば見送り可 |
| **nits** | 些末な指摘。対応は任意（命名の微調整、コメントの改善） |
| **question** | 質問・確認。意図や背景の確認 |
| **suggestion** | 改善提案。より良いアプローチの提示 |

#### 出力フォーマット

```markdown
## Code Review

対象: [PR番号 or ブランチ名]
変更ファイル数: [N]

### 総評

[変更全体に対する1-2文の評価。設計上の問題があれば先に明記する]

### 指摘事項

#### [must] エラーが握りつぶされている
- 箇所: `path/to/file.go:42`
- 問題: `err` が返されずに無視されている。このエラーが発生した場合、後続の処理が不正なデータで続行される
- 改善案: `err` を呼び出し元に返す。ログ出力だけでは不十分

#### [should] 不要なデータベースクエリ
- 箇所: `path/to/service.go:85-92`
- 問題: ループ内で毎回クエリを発行しており、N+1問題が発生する
- 改善案: ループ外で一括取得し、mapで参照する

#### [nits] 変数名の改善
- 箇所: `path/to/handler.go:15`
- 問題: `d` は意図が不明確
- 改善案: `duration` に変更
```

## レビューの原則

- **建設的に伝える**: 問題の指摘だけでなく、具体的な改善案を提示する。「ダメ」ではなく「こうすればより良い」と伝える
- **理由を添える**: 指摘の根拠を示す。「なぜダメなのか」が分からない指摘は受け入れにくい
- **良い点も認める**: 優れた設計判断や工夫が見られる場合は積極的に言及する。レビューは問題を見つけるだけの場ではない
- **完璧を求めない**: 「この変更がコードベースを改善しているか」が判断基準。レビュアーの好みを押し付けない
