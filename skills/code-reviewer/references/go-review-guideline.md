# Go レビューガイドライン

Goプロジェクトのレビュー時に、汎用ガイドライン（`review-guideline.md`）と併せて参照する。

## 1. Effective Go 準拠チェック

### 1.1 命名規則
- [ ] **パッケージ名**: 短く、明確で、小文字のみ使用
- [ ] **関数名**: MixedCapsを使用、アクロニムは統一（例: `ServeHTTP`）
- [ ] **変数名**: 短いスコープでは短い名前、長いスコープでは説明的な名前
  - [ ] 複数英単語の頭文字を抜き出した変数名は、広くエンジニア間で知られているものに限定する
- [ ] **定数**: 適切な命名とグループ化

### 1.2 制御構造
- [ ] **if文**: 初期化付きif文の活用、早期リターンの使用
- [ ] **for文**: range句の適切な使用、不要な変数の回避
- [ ] **switch文**: 型スイッチの適切な使用、fallthroughの明示的な使用

### 1.3 関数
- [ ] **戻り値**: 名前付き戻り値の適切な使用
- [ ] **エラーハンドリング**: `error`型の適切な使用、カスタムエラーの実装
- [ ] **defer文**: リソースの適切な解放

### 1.4 データ構造
- [ ] **スライス**: 適切な容量設定、nilスライスと空スライスの区別
- [ ] **マップ**: 適切な初期化、存在チェック
- [ ] **ポインタ**: 必要性の検討、nilポインタの安全な処理

### 1.5 インターフェース
- [ ] **小さなインターフェース**: 単一責任の原則に従っているか？
- [ ] **暗黙的実装**: 明示的な`implements`宣言を避けているか？
- [ ] **標準インターフェース**: `io.Reader`、`fmt.Stringer`等の活用

## 2. Go言語固有の確認

- [ ] **ゼロ値**: 構造体のゼロ値が有効な状態か？
- [ ] **メモリ管理**: 不要なメモリ使用や、スライス/マップの適切な初期化がされているか？
- [ ] **並行処理**: Goroutineやチャネルの使用が適切か？

## 3. Go Team 推奨事項

### 3.1 コードスタイル
- [ ] **gofmt**: コードフォーマットの統一
- [ ] **go vet**: 静的解析ツールの警告対応
- [ ] **golint**: リンターの推奨事項への対応
- [ ] **go mod**: モジュール管理の適切な使用

### 3.2 並行処理
- [ ] **チャネル**: 適切なチャネルの使用（バッファサイズ、クローズ処理）
- [ ] **sync**: MutexやOnceの適切な使用
- [ ] **context**: コンテキストの適切な伝播

## 4. テスト（Go固有）

- [ ] **テスト名**: `TestXxx`形式で説明的な名前
- [ ] **ベンチマーク**: パフォーマンス重要な部分のベンチマーク
- [ ] **Godoc**: 公開される関数・型・パッケージのドキュメント
- [ ] **Example関数**: 複雑な機能のExample関数
