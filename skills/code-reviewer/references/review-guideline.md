# レビューガイドライン

## 基本方針

**Google's Code Review Guidelines** に準拠して厳格にレビューする。コードの品質向上を最優先とし、妥協のないレビューを実施する。

## 1. 設計レビュー（Design Review）

### 1.1 設計の妥当性
- [ ] **機能の必要性**: この変更は本当に必要か？仕様を満たしているか？
- [ ] **設計の適切性**: 実装がシステム全体の設計に適合しているか？
- [ ] **複雑度**: 実装が必要以上に複雑になっていないか？
- [ ] **将来の拡張性**: 将来的な要件変更に対応できる設計か？
- [ ] **Over-engineering**: 現在の要件に対して過度に汎用的な設計になっていないか？

### 1.2 複雑さ（Complexity）
- [ ] **認知的複雑さ**: 開発者が理解するのに時間がかかりすぎないか？
- [ ] **関数の複雑さ**: 1つの関数で複数の責任を持ちすぎていないか？
- [ ] **ネストの深さ**: if文やfor文のネストが深すぎないか？
- [ ] **抽象化レベル**: 適切な抽象化レベルが保たれているか？
- [ ] **サイクロマティック複雑度**: 制御フローが複雑すぎないか？
- [ ] **依存関係**: 不必要な依存関係がないか？モジュール間の結合度は適切か？

### 1.3 一貫性（Consistency）
- [ ] **コーディングスタイル**: プロジェクト全体のコーディングスタイルと一致しているか？
- [ ] **命名規則**: 既存コードと同じ命名パターンを使用しているか？
- [ ] **設計パターン**: 既存の設計パターンや慣習に従っているか？
- [ ] **エラーハンドリングパターン**: プロジェクト全体で統一されたエラーハンドリングか？
- [ ] **APIの一貫性**: 類似の機能を持つ他のAPIと一貫したインターフェースか？
- [ ] **ファイル構成**: ディレクトリ構造やファイルの配置が既存の規則に従っているか？

### 1.4 APIの設計
- [ ] **命名**: 関数名・変数名が言語の命名規則に従っているか？
- [ ] **可読性**: APIが直感的で理解しやすいか？
- [ ] **一貫性**: 既存のコードベースと命名規則や設計パターンが一致しているか？

### 1.5 ドキュメント（Documentation）
- [ ] **ドキュメントコメント**: 公開される関数・型に適切なドキュメントコメントがあるか？
- [ ] **コメントの品質**: コメントが「なぜ」に焦点を当て、「何を」ではなく「理由」を説明しているか？
- [ ] **使用例**: 複雑なAPIには使用例やサンプルコードが含まれているか？
- [ ] **TODO/FIXMEコメント**: 一時的なコメントに適切な期限や担当者が記載されているか？
- [ ] **API変更の文書化**: 破壊的変更や非推奨化について適切に文書化されているか？
- [ ] **コードの意図**: 複雑なロジックやアルゴリズムの意図が明確に説明されているか？
- [ ] **制約と前提条件**: 関数の前提条件や制約が明確に記載されているか？
- [ ] **エラー条件**: どのような場合にエラーが発生するかが文書化されているか？

## 2. 機能性レビュー（Functionality Review）

### 2.1 動作確認
- [ ] **意図した動作**: コードが開発者の意図した通りに動作するか？
- [ ] **エッジケース**: 境界値や例外的な入力に対する処理が適切か？
- [ ] **エラーハンドリング**: 言語の慣習的なエラーハンドリングパターンを使用しているか？
- [ ] **パフォーマンス**: 明らかなパフォーマンス問題がないか？

## 3. パフォーマンスとメモリ効率

### 3.1 メモリ使用量
- [ ] **不要なアロケーション**: 避けられるメモリ確保がないか？
- [ ] **オブジェクトの再利用**: プールなどの再利用パターンの検討
- [ ] **コレクションの初期化**: 予想サイズが分かる場合の容量指定

### 3.2 計算量
- [ ] **アルゴリズム効率**: より効率的なアルゴリズムはないか？
- [ ] **データ構造選択**: 用途に最適なデータ構造の選択

## 4. テスト

- [ ] **カバレッジ**: 重要な機能のテストが含まれているか？
- [ ] **テスト名**: 説明的で振る舞いを表す名前か？
- [ ] **テーブル駆動テスト**: 複数のケースがある場合の適切な実装
- [ ] **可読性**: DAMPの原則に従ったテストコードか？

## 5. セキュリティ

- [ ] **入力検証**: 外部入力の適切な検証
- [ ] **SQLインジェクション**: データベースクエリの安全性
- [ ] **認証・認可**: 適切な権限チェック
- [ ] **機密情報**: ログやエラーメッセージでの情報漏洩防止

## 6. レビューの進め方

### 6.1 重要度の順序
1. **設計上の問題**: 最優先で指摘
2. **機能の不具合**: 動作に影響する問題
3. **パフォーマンス**: 明らかなボトルネック
4. **可読性**: コードの理解しやすさ
5. **スタイル**: 一貫性の問題

### 6.2 レビューコメントのラベル

各レビューコメントの先頭に以下のラベルを付与し、指摘の重要度を明確にする。

| ラベル | バッジ | 用途 |
|---|---|---|
| must | ![must](https://img.shields.io/badge/security-must-E74C3C) | 修正必須。マージ前に対応が必要な問題（バグ、セキュリティ、設計上の欠陥） |
| should | ![should](https://img.shields.io/badge/consistency-should-F39C12) | 修正推奨。対応が望ましいが、理由があれば見送り可（可読性、パフォーマンス改善） |
| nits | ![nits](https://img.shields.io/badge/docs-nits-3498DB) | 些末な指摘。対応は任意（命名の微調整、コメントの改善、フォーマット） |
| question | ![question](https://img.shields.io/badge/specification-question-9B59B6) | 質問・確認。意図や背景の確認（「なぜこの実装にしたのか？」） |
| suggestion | ![suggestion](https://img.shields.io/badge/complexity-suggestion-2ECC71) | 改善提案。より良いアプローチの提示（「こういう方法もある」） |

**使用例:**

```markdown
![must](https://img.shields.io/badge/consistency-must-E74C3C)

エラーが無視されています。呼び出し元に返すか、適切にハンドリングしてください。
```

```markdown
![nits](https://img.shields.io/badge/naming-nits-3498DB)

変数名`d`は`duration`の方が意図が伝わりやすいです。
```

### 6.3 フィードバック方法
- [ ] **建設的**: 問題の指摘だけでなく、改善案も提示
- [ ] **具体的**: 曖昧な表現を避け、具体的な修正方法を示唆
- [ ] **ラベル付け**: 6.2のラベルを必ず付与し、重要度を明確にする
- [ ] **学習機会**: 説明することで知識を共有

## 7. 承認基準

### 7.1 承認可能な条件
- [ ] 設計が適切で、要件を満たしている
- [ ] 明らかなバグや設計上の問題がない
- [ ] 言語の慣習に従っている
- [ ] 既存のコードベースと一貫性がある
- [ ] 適切なテストが含まれている
- [ ] パフォーマンスに問題がない

### 7.2 承認しない条件
- [ ] 設計上の根本的な問題がある
- [ ] 重要な機能にテストがない
- [ ] セキュリティ上の問題がある
- [ ] 明らかなパフォーマンス問題がある
- [ ] 言語の基本的な慣習に反している

**このチェックリストをすべて満たすまで、PRの承認は行わないこと。**
